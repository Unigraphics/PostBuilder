What’s New in Post Builder Version 6.0

 

Summary

·        UDE Editor – This module was first released in Post Builder version 5.0.3. It provides an integrated environment for the users to modify and create Machine Control events (MCE) and User Defined cycles (UDC).

·        Localization – Windows version of Post Builder 6.0 supports languages of 10 locales as NX does.

·        VNC (Virtual NC Controller) Creator – More graphical user-interface elements have been implemented to facilitate the creation of machine tool simulation drivers.

·        Other Changes & Enhancements

UDE Editor

·        UDE Editor can be enabled for creating a new post or editing an existing post.

·        The enabling switch is a one-time activation only for the post in question. It automatically resets back to “As Saved” when the post in process is closed.

·        Any post created or saved with UDE enabled (Yes) will be opened with UDE being enabled and vice verse; unless the switch is purposely set otherwise before the post is opened.

·        When creating a new post, “As Saved” options functions the same as the “No” option.

The New Post dialog also provides “Enable UDE Editor” toggle allowing the users to activate UDE functionalities for the post to be created.

 

There are 2 classes of CAM objects implemented using UDE mechanism:

-        Machine Control Events (MCE)

-        User-Defined Cycles (UDC)

When a post is created with the UDE editing capability enabled, additional functionalities relevant to the editing and creation of Machine Control events and User-Defined cycles will become available.

Only MCE with corresponding definition in the CDL file (ude.cdl) will display a UDE parameters dialog mimicking the dialog presented in NX/CAM for the MCE in question.

Machine Control Events are presented in 2 categories:

-        System MCE (in light blue)

-        Non-System (Custom) MCE (in green) – CAM provides a set of default. The users can always add new ones as desired.

User-Defined Cycles are also presented in 2 categories:

-        System UDC (in light blue) – They are not defined in the ude.cdl unless new parameters have been added.

-        Non-System UDC (in green) – By default, none exists in ude.cdl; user will add new cycles.

Two types of parameters will be presented on a UDE dialog:

-        System parameters are displayed in cyan; they are read only and not deletable.

-        Custom parameters are displayed in light orange.

 

Handlers for add-on non-system MCE can be constructed the same fashion (drag & drop) as the system MCE.

 

UDE definitions can be previewed on the “Output Settings à Post Files Preview à User Defined Events” page; whereas UDE handlers can be previewed on the “Output Settings à Post Files Preview à Event Handlers” page.

 

UDE definitions & handlers defined for a post can be inherited by another post (NC Data Definitions à Other Data Elements). This mechanism is employed by the on-machine probing project to deliver the definitions and handlers of the probing cycles.

Users can switch to any of the installed languages before creating or opening a post. The choice of language at the time when a Post Builder session has ended will be remembered for the resumption of next session.

 

Users can also use their own language file for localization. Using “my_language.tcl” as the example, she can define the variable “gPB(my_language,LABEL) as the label to be displayed on the “Language” pull-down menu. In this example, “my_language” should be used as the locale name throughout the entire translation file, including gPB(my_language,LABEL), as indicated in the excerpt below. Unicode utf-8 encoding should be used.

##################  DO NOT CHANGE ANYTHING ABOVE THIS LINE ###################

 

if { [info exists gPB(LANG_TEST)] } {

 

  #+++++++++++++++++++++++++++++++++++++++++++

  # Define label in Unicode for this language

  #+++++++++++++++++++++++++++++++++++++++++++

   set gPB(my_language,LABEL) "我的语言"

 

  #++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++

  # Return "1" when this language file is ready to be deployed,

  # otherwise return "0".

  #++++++++++++++++++++++++++++++++++++++++++++++++++++++++++++

 

return 1

}

VNC Creator

Many graphical elements & interactive dialogs have been implemented for the creation of Virtual NC Controllers machine tool simulation.

 

This environment not only enables the users to create and modify the virtual controllers easily; but it also provides a mechanism to prepare the NC program setup data (fixture offsets & tool assignments for now) for external NC code file simulation using the VNC created by Post Builder.

 

vnc_base_v600_tcl.txt

New version of the encrypted VNC base file is included in this release of Post Builder. This file will also be released with NX6.

Machine tool simulation drivers created with this version of Post Builder will require this VNC base file to work. The users of older NX or UG releases need to copy this file from the POSTBUILD directory to the desired UGII_CAM_POST_DIR directory.

Other Changes & Enhancements

·        Post Files Preview page has been moved under the Output Settings group to make room for VNC Creator.

·        Formatted M/G Code Tables – Formatted M & G codes are displayed with specified address leaders for better & clear representation of the NC instructions to be output.

·        XZC-Mill post will be created with MOM_rotate handler (for Rotate UDE) included by default.

·        The problem with the efficiency of saving posts in Post Builder that deteriorates over time has been fixed.

New Custom Commands

1.      PB_CMD_fix_RAPID_SET

This custom command can be added to the Start of Program event to fix the problem with the legacy command that does not account for the –X or –Y directions. It also allows the users to temporarily switch off the work plane change when the spindle axis is not along a principal direction.  PB_CMD_restore_work_plane_change can then be called at the end of the Rapid Move event to restore the setting of work plane change.

2.      PB_CMD_restore_work_plane_change

(As described above)

3.      PB_CMD_set_cycle_plane

This command has been improved to better determine a proper principal axis by using the information in mom_spindle_axis vector.

4.      PB_CMD_abort_event

This command can be used to abort an event according to the condition indicated by the global variable mom_sys_abort_next_event. Linear, Circular, Rapid and Cycle move events all have this trigger built-in by default. As desired, the users can also use this command to abort any other events.

5.      PB_CMD_before_output

This command can be used to manipulate the NC data before it finally gets output to a file. This command, when present, is automatically executed by the MOM_before_output handler. Framework of the command is provided in this release of Post Builder.

As of PB6.0, users should never overload the MOM_before_output handler itself. All customization should be done in PB_CMD_before_output!

Enhancements for Windows version only

1.      External text editor – Users can link an external text editor (via environment variable EDITOR) to the Custom Command page. This functionality has been implemented since Post Builder version 5.0.1.

2.      Color coded text is used in the custom command and the file preview pages.

3.      Mouse wheel support

Legacy Posts Conversion

Pre V3.4 Mill Posts

Preserving New IKS Parameters

In Post Builder 3.3 or earlier, custom commands such as “PB_CMD_init_new_iks” were used to enable the new IKS functionality. Parameters specified in any of these custom commands will be extracted and preserved with the posts when saved in this version of Post Builder.  Interactively, these parameters will be presented in the appropriate dialogs when a post is open. However, the parameters will not be preserved, if the new IKS service was not enabled (mom_kin_iks_usage = 0) in the custom command. Default values for the new IKS parameters will then be presented instead.

These legacy custom commands will no longer be needed but will be retained with the posts for user’s reference. They may be deleted manually.

 

Swapping Rotary Axes of Dual-Head Posts

The assignments of rotary axes in all dual-head 5-axis mill posts created before this version of Post Builder were intentionally switched to accommodate certain short comings with the legacy inverse kinematics solver.  This erroneous setting will be corrected when a post is saved with this version of Post Builder.

Kinematic parameters for the 4th rotary axis will be swapped with that of the 5th axis.

 

PB_CMD_revert_dual_head_kin_vars

This custom command will be provided for all 5-axis mill posts and used to swap the kinematic parameters of the dual rotary heads. This custom command should also be called whenever the new IKS is disabled.

 

VNC of Dual Rotary Heads

Prior to NX4, the NC axes assignment in a VNC file is done in a custom command. There’s no easy way to carry out an automatic conversion. The users will have to manually edit the custom command.

 

Legacy Posts Using ugpadvkins.dll

New IKS service will be disabled for dual-head mill posts using “ugpadvkins” shared library. The kinematic parameters assigned for the dual rotary heads will be swapped as described above.

Pre V3.3 Lathe Posts

Custom command PB_CMD_output_spindle used by any legacy lathe posts to output a block for setting the maximum spindle speed needs to be modified as follows:

 

#=============================================================

proc PB_CMD_output_spindle { } {

#=============================================================

  global mom_spindle_mode

  global spindle_is_out

  global mom_spindle_maximum_rpm

 

   if {![info exists spindle_is_out]} {

 

      if {$mom_spindle_mode == "RPM"} {

         MOM_force once M_spindle S G_spin

         MOM_do_template spindle_rpm

      } elseif {$mom_spindle_mode == "SFM" || $mom_spindle_mode == "SMM"} {

         MOM_force once M_spindle S G G_spin

 

        # The following line has been changed.

      if {$mom_spindle_maximum_rpm > 0} {

         MOM_do_template spindle_max_rpm

      }  

 

         MOM_do_template spindle_css

      }

      set spindle_is_out 1

   }

}

Pre V3.1 Multi-Axis Posts

The option of the Axis Limit Violation Handling for a rotary axis specifies how the postprocessor behaves when the system reaches a rotary axis limit.  The option can be of

·        Warning means that the postprocessor outputs a message to the warning file, but takes no action to correct the axis limit violation.

·        Retract/Re-engage means that the postprocessor generates N/C codes that enable machine-tool to retract to clearance geometry (plane, cylinder or sphere), reposition the rotary axis to a valid position if possible, re-engage to a specified distance above the part, feed to the previous position where it retracted from and then continue moving to the programmed destination.  The feed rate for the retraction will be at the retract feed rate, the feed rate for the move from the clearance back to the re-engage point will be at the approach feed rate and the move from the re-engage point into the part will be at the engage feed rate.

 

Parameters for this control are specified in the custom command PB_CMD_init_rotary under Program & Tool Path -> Custom Command.  As depicted in the graphic below, you can modify the variables mom_kin_retract_type, mom_kin_retract_distance and mom_kin_reengage_distance to configure how this functionality behaves.

 

The parameter mom_kin_retract_plane used in previous releases of Post Builder has been replaced by mom_kin_retract_distance.  PLANE used as one of the options of mom_kin_retract_type in some pre-release versions of v3.1 has been replaced by SURFACE.  After you have saved your legacy posts in Post Builder v3.1, unless you make changes in the custom command PB_CMD_init_rotary, the value of mom_kin_retract_plane will be transferred to mom_kin_retract_distance and PLANE will be interpreted as SURFACE automatically.

 

Pre V3.1 XZC Mill & Simple Mill-Turn Posts

Prior to version 3.1, C axis limits (Machine Tool -> Rotary Axis) of a XZC mill or simple mill-turn post were not observed.

 

No checks were made against the values that were on the dialog for the rotary axis limits.  Only the limits of the fourth_axis Address –9999.9999 ~ 9999.9999 were used.  Starting from version 3.1 the actual limits will be enforced.

 

Example of a post with rotary axis limits of 0 ~ 360:

 

   V3.0.1

G81 X1 Z1 R2.1 C180.0

C260.0

C340.0

C420.0

C500.0

 

   V3.1

G81 X1 Z1 R2.1 C180.0

C260.0

C340.0

C60.0

C140.0

 

When saving a legacy XZC or simple mill-turn post in Post Builder v3.1, if you need to preserve the effect of having the limits of the rotary axis being -9999.9999 ~ 9999.9999, you must change the values on the dialog accordingly.

 

Pre V3.0 XZC Simple Mill-Turn Posts

The information in the custom command PB_CMD_init_mill_turn and PB_CMD_init_mill_xzc will be converted to dialog items on the Machine Tool pages and removed.   The custom commands PB_CMD_start_of_mill, PB_CMD_end_of_mill, PB_CMD_start_of_turn and PB_CMD_end_of_turn have been converted into event handlers.  You can now edit the event handlers on the Linked Posts page.

 

Pre V3.0 Linked Posts

Custom command PB_CMD_init_multiple_post of all legacy posts has been converted to dialog items on the Linked Posts page and removed.    The custom commands used to start and end each head have also been converted into event handlers.  You can now edit these event handlers on the linked posts page.

 

Pre V2.0.2 XZC Mill & Simple Mill-Turn Posts

You will need to follow the following steps to rectify existing XZC mill posts created in Post Builder version 2.0:

1.     Create a new XZC mill post using the existing post as the “controller” post.  Make sure the new post has the same output units as the existing ones.

 

2.     Transfer your changes of the variables, if any, from the custom commands PB_CMD_kin_mill_turn_initialize and PB_CMD_kin_mill_xzc_init to PB_CMD_init_mill_turn and PB_CMD_init_mill_xzc respectively.

 

3.     Add a line of code containing the “return” (without quotes) statement into the beginning of both PB_CMD_kin_mill_turn_initialize and PB_CMD_kin_mill_xzc_init commands.

 

4.     Examine and save the post.
